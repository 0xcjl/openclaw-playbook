# 我是如何让 AI Agent 理解模糊需求的：三层任务路由实践

> 作者：Jialin | 基于 OpenClaw 的实践总结

---

在使用 OpenClaw 构建多 Agent 系统的过程中，我遇到了一个核心问题：

**用户说「帮我看看这个项目」，我该派哪个 Agent 去处理？**

经过几个月的实践和优化，我总结出了一套三层任务路由方法。今天分享给大家。

---

## 问题的本质

人类表达需求时，往往是：
- **模糊的**：「帮我看看这个」
- **隐含的**：「这个工具好用吗」（实际想要对比方案）
- **多层次的**：「做个活动」（需要策略 + 内容 + 执行）

如果只靠关键词匹配，很容易派错 Agent，或者让用户重复描述需求。

我需要一套机制，让系统在用户表达不清晰的时候，也能理解真实意图。

---

## 我的解决方案：三层决策链

### 第一层：关键词初筛（快速排除）

这是最基础的一层，用于快速排除明显不匹配的 Agent。

**实现方式**：
- 扫描用户输入中的关键词
- 「写文章」→ creator Agent
- 「调研」→ master Agent  
- 「写代码」→ dev Agent

**优点**：快
**缺点**：容易被表面词汇误导

**例子**：
用户说「帮我看看这个工具怎么样」，关键词里没有「调研」，但不应该直接回复「好用/不好用」。

这一层只是初筛，不做最终决策。

---

### 第二层：语义理解（识别任务本质）

这一层不看字面意思，而是理解任务的本质。

**实现方式**：
让 LLM 理解这句话背后的真实意图，而不是表面词汇。

**案例**：

| 用户输入 | 表面关键词 | 本质意图 | 路由决策 |
|---------|-----------|---------|---------|
| 「帮我看看这个项目的竞争对手」 | 无「调研」 | 市场分析 | master Agent |
| 「这个功能怎么实现比较好」 | 无「写代码」 | 技术方案 | dev Agent |
| 「帮我想个宣传文案」 | 无「写文章」 | 内容创作 | creator Agent |

这一层已经开始理解「意图」，而不只是「词汇」。

---

### 第三层：LLM 深度推理（最高级）

这是我认为最关键的一层，也是 AI Agent 和传统 bot 本质不同的地方。

**核心能力**：

**① 上下文推理**
结合对话历史理解真实意图。

例如：用户之前说「调研 MailboxKit」，现在问「和它类似的还有哪些」，我不只回答产品列表，而是���对比分析。

**② 隐含需求识别**
识别任务背后的真实目标。

「这个工具好用吗」→ 实际需要的是「对比分析 + 推荐方案」，而不是「好/不好」两个字。

**③ 任务拆解能力**
判断是否需要多个 Agent 协作。

「做一个营销活动」→ 可能需要：
- master Agent（定策略）
- creator Agent（出内容）
- marketing Agent（执行落地）

**④ 目标对齐**
执行过程中持续锚定「用户最终想要什么」，防止在技术细节里迷路。

**⑤ 不确定性处理**
当任务模糊时，主动确认而不是猜测。

「帮我处理这个」→ 我会先问「你希望如何处理」，而不是自作主张。

---

## 实际工作流

```
收到任务
    ↓
关键词初筛（快速排除明显不匹配）
    ↓
语义理解（识别任务本质）
    ↓
LLM 深度分析（结合上下文 + 隐含需求 + 复杂度）
    ↓
决策：单 Agent / 多 Agent / 需要先确认
    ↓
派发 / 确认 / 执行
```

## 实践中的关键发现

### 发现 1：不要依赖关键词

早期版本中，我尝试用关键词词库来路由任务。结果发现：
- 用户表达方式千变万化
- 维护词库成本极高
- 容易漏掉边缘 case

**改进**：让 LLM 理解意图，而不是匹配关键词。

### 发现 2：主动确认 > 猜测

当任务模糊时，有两种选择：
- A）猜一个最可能的方向
- B）主动问用户

我的实践结论：**B 更好**。

虽然多了一轮交互，但避免了「做了半天发现方向错了」的浪费。

### 发现 3：上下文是金矿

用户很少在一句话里说清楚所有信息。但如果结合对话历史，很多隐含需求就能推断出来。

**例子**：
- 用户之前问过「MailboxKit 的定价」
- 现在问「有没有更便宜的替代品」
- 系统应该理解：这是在做成本优化决策，需要对比分析

## 对其他开发者的建议

如果你也在构建多 Agent 系统，这些经验可能有用：

### 1. 不要过度依赖规则引擎
规则引擎适合处理明确的、可枚举的场景。但用户需求往往是模糊的、多变的。

**建议**：让 LLM 做决策，而不是写 if-else。

### 2. 设计好 Agent 的职责边界
如果 Agent 职责不清晰，路由决策会很难做。

**建议**：每个 Agent 应该有清晰的「擅长领域」和「不擅长领域」。

### 3. 记录失败 case
每次路由错误，都是优化的机会。

**建议**：建立一个「路由失败日志」，定期回顾和优化。

### 4. 用户反馈是最好的测试
再完善的逻辑，也比不上真实用户的反馈。

**建议**：在早期版本中，主动问用户「我理解对了吗」，收集反馈。

## 未解决的问题

这套方法还有一些局限：

1. **多 Agent 协作的优先级问题**
   当多个 Agent 都能处理时，该选哪个？

2. **任务拆解的粒度问题**
   什么时候该拆成多个子任务，什么时候该合并处理？

3. **成本与准确度的平衡**
   每次都用 LLM 深度推理，成本会不会太高？

这些问题我还在探索中，欢迎交流。

---

## 总结

这套三层任务路由方法，是我在使用 OpenClaw 过程中实践出来的。

核心思路：
- **第一层**：快速初筛（关键词）
- **第二层**：理解本质（语义）
- **第三层**：深度推理（LLM）

不是完美方案，但在我的场景下效果不错。

如果你也在做类似的事情，希望这些经验对你有用。
